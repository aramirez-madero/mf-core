Quiero construir un sistema llamado **MF Core** para Madero Factoring.

MF Core no debe ser una herramienta aislada. Debe diseñarse como una **plataforma modular, escalable y trazable**, donde cada módulo pueda alimentar a otros módulos futuros.

El primer módulo será un **Generador de Anexos y Control Histórico**, basado principalmente en el archivo exportado de **CAVALI**.

El MVP inicial debe enfocarse en:

1. Cargar el archivo Excel/CSV descargado de CAVALI.
2. Leer la información base desde ese archivo.
3. Cruzar la información con bases maestras internas.
4. Aplicar reglas de validación.
5. Aplicar la regla de **Participante Origen**.
6. Generar uno de los 3 tipos de anexos existentes.
7. Generar un Excel final con estructura similar a la hoja **HISTORICO** del archivo actual de control.
8. Guardar los registros finales en una base llamada **Control**.
9. Guardar historial, estado y versión de cada proceso.
10. Dejar el sistema preparado para crecer luego hacia desembolsos, cobranzas, liquidaciones, tesorería, portal cliente, portal inversionista, documentos, reportes e integraciones.

---

# 1. Enfoque general del sistema

La fuente principal de información será el archivo exportado desde **CAVALI**.

A partir de ese archivo se debe generar:

* La información necesaria para crear los anexos.
* Un Excel final con hoja **HISTORICO**.
* Registros en la tabla/base **Control**.
* Trazabilidad completa del proceso.
* Datos reutilizables para módulos futuros.

Los PDF/XML originales de las facturas no se guardarán en MF Core, porque esa información ya está disponible en otro sistema. MF Core solo trabajará con la información necesaria para crear anexos, validar datos y registrar el resultado.

El flujo general será:

Archivo CAVALI
↓
MF Core lee y normaliza los datos
↓
MF Core cruza con bases maestras internas
↓
MF Core valida información
↓
MF Core genera anexo
↓
MF Core genera Excel tipo HISTORICO
↓
MF Core guarda registros en Control
↓
MF Core guarda historial y trazabilidad
↓
Otros módulos podrán reutilizar esa información

# 5. Normalización de datos

Antes de cruzar información, el sistema debe normalizar:

* RUC: quitar espacios, guiones y caracteres innecesarios.
* Fechas: convertir a formato estándar `YYYY-MM-DD`.
* Montos: convertir texto a número decimal.
* Moneda: homologar valores como `S/`, `PEN`, `SOLES`, `USD`, `$`.
* Serie: limpiar espacios.
* Numeración: mantener ceros si corresponde.
* Factura: construir con Serie + Numeración.
* Razón social: limpiar dobles espacios y caracteres extraños.
* Código de Participante Origen: convertir a texto o número limpio.

Ejemplo:

Si CAVALI trae:

Serie: E001
Numeración: 1613

El campo Factura debe quedar:

E001-1613

Si el formato del HISTORICO requiere prefijo como `FA-`, dejarlo parametrizable.

# 7. Regla importante: Participante Origen

El archivo de CAVALI tiene un campo llamado **Participante Origen**.

La regla debe ser:

## Caso 1: Participante Origen = 841

Si **Participante Origen = 841**, significa que la operación es directa de Madero Factoring.

Entonces:

* En el anexo solo deben salir los datos del proveedor.
* No debe agregarse información de otro participante externo.
* En el HISTORICO, el campo `Originador` puede marcarse como `MF` o `Madero Factoring`.
* El campo `requiere_datos_participante` debe ser `No`.

## Caso 2: Participante Origen diferente de 841

Si **Participante Origen ≠ 841**, significa que interviene otro participante/factor/originador externo.

Entonces:

* El sistema debe buscar ese código en `maestro_participantes_origen`.
* En el anexo deben salir los datos del proveedor y también los datos del participante origen.
* En el HISTORICO, el campo `Originador` debe salir según el participante encontrado en la base maestra.
* El campo `requiere_datos_participante` debe ser `Sí`.

## Validación obligatoria

Si el código del Participante Origen es diferente de 841 y no existe en `maestro_participantes_origen`, el sistema NO debe permitir generar el anexo ni el registro final.

Debe marcar el registro como observado con el mensaje:

“Participante Origen no encontrado en maestro. Debe registrar este participante antes de generar el anexo.”

# 18. Flujo del MVP

El flujo esperado es:

1. Usuario entra a MF Core.
2. Usuario carga archivo CAVALI.
3. MF Core lee el archivo.
4. MF Core normaliza fechas, montos, moneda, RUC, serie y numeración.
5. MF Core cruza proveedor por RUC.
6. MF Core cruza adquirente por RUC.
7. MF Core cruza Participante Origen con maestro_participantes_origen.
8. MF Core aplica regla de Participante Origen = 841.
9. MF Core arma los campos de la estructura HISTORICO.
10. MF Core muestra vista previa de validación.
11. Usuario corrige o completa maestros si hay observaciones.
12. Usuario confirma registros válidos.
13. MF Core genera el anexo correspondiente.
14. MF Core genera Excel final con hoja HISTORICO.
15. MF Core guarda registros en tabla Control.
16. MF Core guarda historial de acción.
17. MF Core muestra resumen final de la carga.

Usar estructura modular:

* Servicio de lectura de CAVALI.
* Servicio de normalización.
* Servicio de cruce con maestros.
* Servicio de validación.
* Servicio de generación de HISTORICO.
* Servicio de generación de anexos.
* Servicio de guardado en Control.
* Servicio de historial.
* Servicio de estados.

No crear una sola función gigante.

Separar la lógica en servicios, por ejemplo:

* `cavaliParserService`
* `normalizationService`
* `masterDataService`
* `validationService`
* `historicoBuilderService`
* `annexGeneratorService`
* `controlService`
* `auditService`
* `documentService`
* `workflowStateService`

* `disbursementService`
* `collectionService`
* `settlementService`
* `investorPortalService`
* `clientPortalService`

Cada módulo debe poder consultar información de módulos anteriores mediante IDs y relaciones.

* Un anexo debe estar relacionado a una operación.

* Una carga CAVALI debe estar relacionada a los registros que generó.
* Un registro de Control debe poder conectarse con anexos, desembolsos, cobranzas y liquidaciones.

# 24. Importante

No construir todo el sistema completo de una vez.

Primero construir el MVP:

* Carga de CAVALI.
* Normalización.
* Cruce con maestros.
* Validación.
* Vista previa.
* Regla Participante Origen.
* Generación de HISTORICO.
* Guardado en Control.
* Historial y trazabilidad.
* Base preparada para generación de anexos.

Luego avanzar con:

* Generación de los 3 anexos.
* Guardado de anexos.
* Estados de envío/firma.
* Desembolsos.
* Cobranzas.
* Liquidaciones.
* Portal cliente.
* Portal inversionista.

Objetivo final:

El MVP debe resolver la generación de HISTORICO y anexos, pero la arquitectura debe permitir que MF Core se convierta progresivamente en una plataforma integral para la gestión operativa de Madero Factoring.

No construir solo para el problema actual. Construir el módulo actual como la primera pieza de una plataforma mayor.
